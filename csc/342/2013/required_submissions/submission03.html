Overview

This week you will write a short program that finds anagrams. An anagram is a word or phrase formed from the letters in another word or phrase. For example, the word slot is an anagram of the word lost. We shall only be interested in single-word anagrams, and not phrase anagrams.

Our goal is to have a program to which we can supply as input a single word, or a single sequence of characters containing no blanks, but which does not have to be an actual word, and the program will then search through a dictionary of words to find and display all words that can be formed from the letters in the input (in other words, to find and display all anagrams of the input).

Steps to Perform

  1. Download the sample executable and the accompanying dictionary file.
  2. Run the program, enter the name of the dictionary file and then enter some words or letter-sequences and have the program search for corresponding anagrams. Here are some to try: 
          list, last, lost, lust, gardenia, grow, grinch, groan, emanate, drawback,
      bacterial, vacate, glare, hate, mental, leap, leapt, slate, name, mate, edit, 
      item, rinse, stripe, live, stance, tuba, came, care, garden, latucoage
    Note that since the word grinch, for example, does not appear in the dictionary we get no results at all. The word grow, on the other hand, has no anagrams, so all we get back when it is the input is the word itself, since it is in the dictionary. We could also input the string rogw and get back grow, since grow is an anagram of that string. This is also illustrated by the last example above, which is not a word.
  3. Continue experimenting until you thoroughly understand the problem and the behavior of the program. Note in particular the error message output if you enter a bad file name for the dictionary.
  4. Write pseudocode for a solution of this problem and revise it until you are convinced that a program that implements your pseudocode will be the one you want.
  5. Translate your pseudocode into C++.
  6. Continue testing your program until you are convinced it works correctly.
  7. Make a final check to ensure your source code is identified, formatted, and documented properly and you have included the self-assessment /* ... */ comment to tell the marker the state of your submission.
  8. Finally, submit the required files over the web.

Additional Notes, Requirements, Specifications and/or Hints (if any)

  1. We said in the Overview above that you would be writing a "short" program, and we meant it, in the following sense. Your submitted source code file must not exceed a page of code, which in Visual Studio is 66 lines. If it does, the marker will deduct a point for each line in excess of this limit. This limitation is designed to make you think seriously about your program design and make maximal use of the STL to accomplish whatever tasks need to be done.
  2. You must include follow the usual style guidelines, with the exception that since the program must be on one page you need not use blank lines to separate "logical chunks" of your code, or comment your code on separate lines. But that means you need to take even more care than usual to write readable code, possibly with help of in-line comments. You must, however, have the usual self-assessment comment, so there is incentive to have a complete and working program so this can be short!
  3. Among the style rules you must follow we mention these in particular: one statement per line; and the rule that requires a brace to be on a line by itself and line up vertically with its matching brace.
  4. You may write your program under the assumption that both the string entered by the user and all dictionary entries are in lower case.
