Overview

This week you will begin a two-part implementation of an ADTBinarySearchTree class for storing character data. We discussed in class the implementation of a template class for the binary search tree, but your implementation will not be a template class. This will make the implementation simpler, but there will still be plenty to do, and the main goal is to make you familiar with binary search trees and operations on them, rather than give you more C++ template programming practice.

For brevity you will name the class BSTChar. A driver is supplied, as well as a shell file (BSTChar_globals_shell.hpp) containing a couple of "busy work" functions (for building the menu and displaying the program information), as well as prototypes for the high-level functions that deal with user interaction. You will fill in the bodies of these high-level prototypes to complete their definitions, and, of course, design and write the BSTChar class in the usual form of a separately compiled .h/.cpp file pair.

You already have some code that you can adapt for use from the sample programs illustrating binary search trees. Much of the remaining code (for this Part 1 and also for Part 2) you have available in pseudocode form from the on-line notes.

The approach in this last two-part submission for the term differs somewhat from previous two-part efforts. You must complete the project by the end of the two weeks, but you get to decide how much and which parts you will complete the first week. Also, the marker can be brutally frank at the end of the first week in his comments about about what you need to do by the end of the second week. This is a double-edged sword, since you could conceivably do nothing the first week and still get a perfect mark. (However, you probably won't be able to do nothing, or almost nothing, the first week and finish the project.) It would be much smarter to do as much as you can the first week, so you have less to do the second week and can fine tune what you have already done. I'll be interested to see how people handle this extra bit of "freedom" in submission preparation.

In any case, whatever mark you get on the second week's submission, when the project should be complete, will apply to both weeks. So, the same mark will be recorded for each week on Blackboard.

Steps to Perform

  1. Copy all of the sample files.
  2. Run the sample executable (which is the complete program, so it is not numbered) and try all of the menu options enough times to get a good feel for how the program behaves.
  3. Give some thought to the order in which you will implement the various things that need implementing.
  4. Design your class.
  5. Write pseudocode for all the global functions for which you have been given the prototypes, as well as all member functions and any helper functions, for your class. Revise your pseudocode until you are convinced that a class that implements your pseudocode will be the one you want.
  6. Decide in what sequence you want to "plug in" and test your code.
  7. Continue testing until the chosen part of the class for this week is complete and you are convinced everything works correctly.
  8. Make sure your source code is identified, formatted, and documented properly, according to our current guidelines.
  9. Finally, submit the required files over the web.
